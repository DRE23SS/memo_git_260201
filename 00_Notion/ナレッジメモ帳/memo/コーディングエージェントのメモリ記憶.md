- LLM が一度に処理できる情報量

  - コンテキストウィンドウ（入出力トークン数）によって決まる

  - 会話やコードが長くなると古い順から忘れる（会話時に考慮しなくなる）

    - →Lost in Middle ※CPU の RAM のイメージ

  - 過去の知識や進捗を保持して取り出せる長期記憶が必要になる

- コーディングエージェントのメモリ管理

  - インデックス化

    - Cursor で使用

    - 情報をベクトル DB に埋め込む

    - 意味的に近い断片を検索して呼び出す

  - プレーンファイル

    - Claude Code や Cline

      - 必要な時にローカルファイルを都度参照

      - 固有のルールファイルやメモリファイルに書き込み使用

- 各エージェントの記憶の置き方

- Cline

  - Memory Bank を使用

    - ./memory-bank の markdown ファイル群

    - それらの扱いを定義する./clinerules のルールファイル群

    - [https://github.com/RPG-fan/Cline-Recursive-Chain-of-Thought-System-CRCT-](https://github.com/RPG-fan/Cline-Recursive-Chain-of-Thought-System-CRCT-)

  ```Python

  your-project/
  └─ memory-bank/
     ├─ projectbrief.md     # 1. 概要・全体像・ゴール
     ├─ productContext.md   # 2. ビジネス価値・UX
     ├─ systemPatterns.md   # 3. アーキテクチャと意思決定
     ├─ techContext.md      # 4. 技術スタックと制約
     ├─ activeContext.md    # 5. いま取り組んでいる作業
     └─ progress.md         # 6. 進捗・既知の課題
  ```

- Claude Code

  - 単一の CLAUDE.md に進捗や方針を記録

  - 配置する階層によってスコープが異なる
    - 一般的なのは、リポジトリ直下に配置

- Kiro

  - 2 つのプレーンファイル

    - Steering 恒常的な知識ファイル

    ```Python
    your-project/
    └─ .kiro/
       └─ steering/
          ├─ product.md     # 製品の目的・主要機能・ビジネス目標
          ├─ tech.md        # 技術スタック・技術的制約
          ├─ structure.md   # ファイル構成・命名規則・アーキテクチャ
    ```

    - Specs 　セッション単位の仕様ファイル

    ```Python

    your-project/
    └─ .kiro/
       └─ specs/
          ├─ session01/
          │   ├─ requirements.md  # ユーザーストーリー・受け入れ基準
          │   ├─ design.md        # 技術アーキテクチャ、シーケンス図、実装上の考慮事項
          │   └─ tasks.md         # 詳細な実装計画・タスクリスト
          ├─ session02/
          │   ├─ requirements.md
    ```
