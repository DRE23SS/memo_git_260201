# Google Antigravity Skills 入門：AIへの指示が伝わらない問題を解決（要約・運用メモ）

## 1) 要約（3点以内）
- 「AIに何をどう伝えるかで迷う」問題を、**/team で起動する“質問駆動ワークフロー”**で解消する、という全体像。
- Antigravityのカスタマイズ要素（Rules / Workflows / Skills）を分けて設計し、**要件定義→設計→実装→レビュー→ドキュメント→振り返り**まで一気通貫にする。
- つまずきがちな点はほぼファイル配置と起動方法（スラッシュコマンド、workflowファイル有無、skillパス）に集約される。

## 2) 前提・仮定（1〜3文）
前提：この記事の内容を、Obsidianで再利用しやすい「運用メモ」として要約する。  
前提：記事中の具体例（ファイル内容など）は著作権配慮のため“そのまま全文転載”はせず、要点＋実務テンプレ（補完）で整理する。

---

## 3) 推奨（優先度付き）
**P0（最短で効く）**
- まずは `/team` を“入口”に固定し、AIに「質問させる設計」に寄せる（自分が完璧な指示を書く前提を捨てる）。

**P1（再現性を上げる）**
- Rules / Workflows / Skills を混ぜずに役割分担する（常時ルール／手動手順／専門知識パッケージ）。

**P2（運用で事故らない）**
- つまずきポイント3つ（起動方法、workflow不在、skills不在）をチェックリスト化して、新プロジェクト開始時に必ず確認する。

---

## 4) 実行手順（ステップ）
1. フォルダに `.agent/workflows/` を用意する（なければ作る）。
2. `/team` で起動する workflow（例：`team.md`）を配置する。
3. workflow が参照する skills（要件定義/設計/レビュー/ドキュメント/振り返り）を、想定パスに作る。
4. `/team` を実行して、Phase 1（要件定義）から順に進むことを確認する。
5. うまくいったら、`.agent/` と `skills/` を“テンプレ資産”として別プロジェクトにコピーして再利用する。

---

## 5) 記事の要点メモ

### 5.1 何を解決する記事か
- 非エンジニア個人開発で起きがちな「AIの解釈がズレる」「要件が揺れる」「命名で止まる」を、**AIが質問して進める**形に変える。

### 5.2 Antigravityの3要素（整理）
- **Rules**：常時適用のルール（例：常に日本語、など）
- **Workflows**：手動で呼ぶ手順（例：`/team`）
- **Skills**：AIが必要に応じて参照する専門知識パック

（補完）運用感としては、「迷いが出る工程をSkills化」「工程の順番と確認をWorkflow化」「ブレる原則をRules化」すると安定しやすい。

### 5.3 `/team` ワークフローの流れ（概念）
- Phase 1: 要件定義（質問で要件を固め、フォルダ名提案→承認→作成）
- Phase 2: 設計（設計書を作る）
- Phase 3: 実装（設計書に沿って実装）
- Phase 4: レビュー（ダブルチェック）
- Phase 5: ドキュメント（READMEや報告資料まで作る）
- Phase 6: 振り返り（次回に学びを残す）
- 各Phaseの終わりに「次へ進む？」確認が入り、途中終了も可能。

### 5.4 つまずきポイント（記事で挙げられている系統）
- 自然言語では起動しない → **スラッシュコマンドで起動する仕様**
- `/team` が見つからない → **workflowファイルが存在しない/配置が違う**
- skillが見つからない → **skillファイルのパスが違う/未作成**

## 6) （補完）Obsidian用：最小テンプレ

### 6.1 プロジェクト開始チェック
- [ ] `/team` のworkflowがある
- [ ] workflowが参照するskillsがある（要件/設計/レビュー/ドキュメント/振り返り）
- [ ] 生成物の出力先（例：`docs/`, `src/`）が想定通り

### 6.2 （補完）workflow設計メモ（コピペ用・概念）
- 目的：`/team` で一気通貫、AIが質問して進める
- 必須要件：
  - Phaseごとの参照skillを明示
  - Phase完了ごとに「次へ進む？」確認
  - 途中終了時は、そこまでの成果物を保存して終了
  - 完了時に成果物一覧を出す

※実際のworkflow本文（YAMLや細かな書式）は元記事の手順に沿って作成するのが安全。
